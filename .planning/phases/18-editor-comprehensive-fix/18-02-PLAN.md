# Phase 18 Wave 2: Button State + Undo/Redo + Sync Guard + Selection Retention

**Phase**: 18
**Wave**: 2 of 3
**Status**: Ready

## Goal

With the Wave 1 selection infrastructure in place, fix four downstream correctness bugs that depend on it: remove the shadow-DOM-blind `queryCommandState` calls from all buttons, scope undo/redo state per editor instance, guard the MutationObserver sync loop against re-entrancy, and verify selection is retained after every style application.

Decisions addressed: D-05, D-06, D-07, D-08.

## Prerequisites

Wave 1 must be complete:
- `safeGetRange(shadowRoot?)` in BrowserCompat.ts is shadow-DOM-aware.
- `getComputedStyles()` in StyleEngine.ts uses `window.getComputedStyle`.
- `saveSelectionAsOffsets` / `restoreSelectionFromOffsets` use editor-root-relative offsets.
- `hasStyleInRange` is correctly detecting styles on semantic elements.

### Task 2.1: Remove queryCommandState from all buttons — replace with StyleEngine detection (D-05)

**Files**: `src/Editor/BoldButton.tsx`, `src/Editor/ItalicButton.tsx`, `src/Editor/UnderlineButton.tsx`, `src/Editor/TextStyleButton.tsx`

**Decision**: D-05

**Description**:
`document.queryCommandState(command)` is called in two places per button:
1. In `handleClick` (BoldButton line 54, ItalicButton line 54): `isActive(document.queryCommandState(command))` — called immediately after applying the style.
2. In `TextStyleButton.tsx` at the exported `updateStylesState` function (line 129-137): called by the `selectionchange` handler in BoldButton, ItalicButton, and TextStyleButton.

`queryCommandState` is shadow-DOM-blind — it always returns `false` for content inside `<wui-editor>` shadow root, so buttons never show as active. After D-02 (Wave 1), `StyleEngine.hasStyleInRange` correctly detects styles on semantic elements using `getComputedStyle`. The fix is to replace `queryCommandState` with `hasStyleInRange` calls.

The `hasStyleInRange` function is currently not exported from `StyleEngine.ts` — it must be exported as part of this task.

A command-to-style-property mapping is needed because buttons use string command names (`"bold"`, `"italic"`, `"underline"`) but `hasStyleInRange` takes CSS property names and values. Define this map once in `TextStyleButton.tsx`.

**Implementation**:

**Step 1 — Export `hasStyleInRange` from StyleEngine.ts**:

Change line 128 in `StyleEngine.ts` from:

```typescript

```

to:

```typescript
export function hasStyleInRange(range: Range, prop: string, value: string): boolean {
```

**Step 2 — Update `TextStyleButton.tsx` `updateStylesState`**:

Replace the exported `updateStylesState` function (lines 129-137) with a version that uses `hasStyleInRange` and the shadow-DOM-aware `safeGetRange`:

```typescript
import { safeGetRange } from './BrowserCompat'
import { hasStyleInRange } from './StyleEngine'

// Map command names to CSS property+value pairs for StyleEngine detection.
// D-05: replaces queryCommandState which is shadow-DOM-blind.
const COMMAND_STYLE_MAP: Record<string, { prop: string; value: string }> = {
    bold: { prop: 'fontWeight', value: 'bold' },
    italic: { prop: 'fontStyle', value: 'italic' },
    underline: { prop: 'textDecorationLine', value: 'underline' },
    strikethrough: { prop: 'textDecorationLine', value: 'line-through' },
}

export const updateStylesState = (
    isActive: Observable<boolean>,
    editor: Observable<HTMLDivElement>,
    command: string
) => {
    const el = $$(editor)
    if (!el || typeof el.contains !== 'function') {
        isActive(false)
        return
    }

    // Only update when editor has focus (same guard as before)
    if (document.activeElement !== el && !el.contains(document.activeElement)) {
        // Editor doesn't have direct focus — check shadow root
        const rootNode = el.getRootNode()
        const shadowRoot = rootNode instanceof ShadowRoot ? rootNode : null
        if (shadowRoot && !shadowRoot.contains(document.activeElement)) {
            isActive(false)
            return
        }
    }

    const rootNode = el.getRootNode()
    const sr = rootNode instanceof ShadowRoot ? rootNode : undefined
    const range = safeGetRange(sr)
    if (!range) {
        isActive(false)
        return
    }

    const entry = COMMAND_STYLE_MAP[command]
    if (!entry) {
        isActive(false)
        return
    }

    try {
        isActive(hasStyleInRange(range, entry.prop, entry.value))
    } catch (e) {
        isActive(false)
    }
}
```

Add the two new imports at the top of `TextStyleButton.tsx`:

### Task 2.3: Editor.tsx — MutationObserver re-entrancy guard on syncChildren (D-07)

**File**: `src/Editor/Editor.tsx`

**Decision**: D-07

**Description**:
`syncChildren()` in `EditorSurface` (lines 139-220) is called by a `MutationObserver` on the host's light DOM. It modifies the shadow DOM `el`, which triggers the second observer (lines 256-263) watching `el` for `saveDo()`. If `saveDo()` triggers Woby reactive updates that write back to the light DOM, `syncChildren()` fires again, creating a feedback cycle.

Add a `let isSyncing = false` guard inside the `useEffect` closure that creates `syncChildren`. This guard is local to the closure so each Editor instance gets its own flag. The guard must be set before any DOM write and cleared in a `finally` block.

The guard must be placed INSIDE the `useEffect(() => {...})` callback that defines `syncChildren`, not at component scope — placement at component scope would be shared across Woby re-render calls.

**Implementation**:

Inside the Light DOM to Shadow DOM Sync `useEffect` (starting at line 123), immediately before the `syncChildren` function definition, add the guard declaration:

```typescript
// D-07: re-entrancy guard prevents MutationObserver feedback cycle.
// MUST be inside useEffect closure (not component scope) — each effect instance owns its flag.
let isSyncing = false
```

Wrap the body of `syncChildren` with the guard:

```typescript
const syncChildren = () => {
    // D-07: prevent re-entrant sync from MutationObserver feedback loops
    if (isSyncing) return
    isSyncing = true
    try {
        // -- existing syncChildren body starts here --

        // Save current selection before clearing content
        const sel = window.getSelection()
        let savedRange: Range | null = null
        // ... (rest of existing code) ...

        // -- existing syncChildren body ends here --
    } finally {
        isSyncing = false
    }
}
```

The entire existing body of `syncChildren` (lines 140-220) moves inside the `try` block. The `isSyncing = false` in `finally` guarantees the flag is cleared even if an exception is thrown during DOM operations.

No other changes to `Editor.tsx` are needed for D-07.

### Task 2.4: StyleEngine — verify selection retention after applyStyle / removeStyle (D-08)

**File**: `src/Editor/StyleEngine.ts`

**Decision**: D-08

**Description**:
After Wave 1 (D-04), `restoreSelectionFromOffsets` now uses editor-root-relative offsets and correctly handles cross-block selections. D-08 is a verification + robustness task: add an assertion log in `applyStyle()` and `removeStyle()` that compares `window.getSelection().toString()` before and after the style operation. This gives the MCP test suite a reliable signal.

Additionally, ensure `applyStyle()` calls `restoreSelectionFromOffsets` unconditionally after DOM mutation (not only when `savedSelection.editorRoot` is truthy). If `findEditorRoot` returns null (editor not found in DOM), log a warning and return — but do not silently skip restoration when the root exists.

The `safeGetSelection()` import already exists at the top of StyleEngine.ts. No new imports needed.

**Implementation**:

In `applyStyle()`, capture the selection text string before the operation and compare after restoration:

```typescript
export function applyStyle(prop: string, value: string): void {
    const range = safeGetRange()
    if (!range) return

    const sel = safeGetSelection()
    if (!sel) return

    // D-08: capture selection text before DOM mutation for post-restoration verification
    const selectionTextBefore = sel.toString()

    // Save original selection as offsets BEFORE any DOM operations
    const savedSelection = saveSelectionAsOffsets(range)

    // ... (existing toggle check and apply logic) ...

    // Normalize after operation (use editorRoot as fallback for cross-block)
    const normalizeTarget = getBlockParent(range.commonAncestorContainer) ?? findEditorRoot(range.commonAncestorContainer)
    if (normalizeTarget) {
        normalizeDOM(normalizeTarget)
    }

    // D-08: restore selection, then verify
    if (savedSelection.editorRoot && savedSelection.startOffset >= 0 && savedSelection.endOffset >= 0) {
        restoreSelectionFromOffsets(savedSelection.editorRoot, savedSelection.startOffset, savedSelection.endOffset)
        const selectionTextAfter = window.getSelection()?.toString() ?? ''
        if (selectionTextBefore !== selectionTextAfter && selectionTextBefore.trim() !== '') {
            console.warn('[applyStyle] Selection text changed after restore. Before:', JSON.stringify(selectionTextBefore), 'After:', JSON.stringify(selectionTextAfter))
        }
    } else if (!savedSelection.editorRoot) {
        console.warn('[applyStyle] Could not find editor root for selection restoration — cursor may be lost')
    }
}
```
